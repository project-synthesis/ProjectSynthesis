# Fully Organic Sub-Domain Vocabulary

**Goal:** Replace the static `_DOMAIN_QUALIFIERS` dict with fully organic LLM-generated vocabulary so all domains — including SaaS, backend, database — discover sub-domains through the same organic pipeline that new domains use.

**Problem:** The static vocabulary in `heuristic_analyzer.py` gates out organic LLM-generated vocabulary for any domain that has a curated entry (`if not domain_qualifiers:` at engine.py:1835). This means:
- SaaS, backend, frontend, database, devops, security, data — all get only their curated keywords
- The curated keywords are too narrow (SaaS gets 0/71 domain_raw hits, only 41/71 via intent_label)
- The `SUB_DOMAIN_QUALIFIER_MIN_KEYWORD_HITS = 2` threshold is too high for Source 1 enrichment
- `signal_keywords` (TF-IDF) are not populated for domains with static vocab, killing Source 3

**Solution:** Delete the static vocabulary entirely. All domains get organic Haiku-generated vocabulary from their cluster labels. The organic vocabulary is cached on domain node metadata and served to the hot path via `DomainSignalLoader`. The enrichment threshold drops to 1 keyword hit. Signal_keywords are populated for all domains.

---

## Architecture

Three components change. Organic vocabulary generation becomes the single, universal method. The other two components (enrichment threshold, TF-IDF signal_keywords) support it by improving signal capture across all three sources of the Phase 5 cascade.

### 1. Vocabulary Generation (Phase 5, warm path)

**Current:** The `if not domain_qualifiers` gate at engine.py:1835 skips LLM generation for any domain with a `_DOMAIN_QUALIFIERS` entry. Only organic/new domains get Haiku-generated vocabulary.

**Change:** Remove the gate. Every domain gets Haiku-generated vocabulary on every Phase 5 run where its cached vocabulary is missing or stale (cluster count changed by 30%+). On the very first Phase 5 cycle after deployment, all domains are stale (no cache exists), so vocabulary generation runs eagerly for all of them before qualifier evaluation begins. This makes the first warm cycle longer (~5-10s per domain for Haiku calls) but immediately productive.

**Vocabulary lifecycle:**
- Generated by Haiku from cluster labels within the domain (existing `generate_qualifier_vocabulary()` function)
- Cached in `cluster_metadata["generated_qualifiers"]` on the domain node
- Refreshed when cluster count changes by 30%+ (existing staleness check)
- Propagated to `DomainSignalLoader` in-memory cache after generation so the hot path sees new vocab immediately

### 2. Hot-Path Enrichment (`_enrich_domain_qualifier`)

**Current:** Reads from the hardcoded `_DOMAIN_QUALIFIERS` dict. Requires `SUB_DOMAIN_QUALIFIER_MIN_KEYWORD_HITS = 2` keyword hits from the prompt text to append a qualifier. Most SaaS prompts only hit 1 keyword, so `domain_raw` stays plain (no qualifier).

**Change:** Refactor to read from `DomainSignalLoader.get_qualifiers(domain)`, which caches the organic vocabulary loaded from domain node metadata. The loader already exists as a process-level singleton — it needs a new `get_qualifiers(domain: str) -> dict[str, list[str]]` method alongside its existing keyword signal methods, plus a `refresh_qualifiers(domain: str, qualifiers: dict)` method for Phase 5 to push updates after generation.

**Threshold:** Lower `SUB_DOMAIN_QUALIFIER_MIN_KEYWORD_HITS` from 2 to 1. The domain is already confirmed (the prompt is classified as "saas" before enrichment runs) — one keyword hit is strong enough signal to append the qualifier. This dramatically improves Source 1 (domain_raw) hit rates.

**Cold-start behavior:** When `DomainSignalLoader` has no cached vocabulary for a domain (fresh deployment, before first Phase 5 cycle), `get_qualifiers()` returns an empty dict. `_enrich_domain_qualifier()` returns the plain domain unchanged (no qualifier appended). This is silent and expected — the health endpoint's `domains_without_vocab` counter surfaces it for operators.

### 3. TF-IDF Signal Keywords for All Domains

**Current:** The SaaS domain node has no `signal_keywords` in its metadata. The `domain_signal_extractor` computes TF-IDF keywords, but the results may not reach all domain nodes — specifically those that had static vocabulary.

**Change:** Ensure `signal_keywords` are computed and stored for every domain node during the signal extraction phase (Phase 4.75 or equivalent). This provides Source 3 fallback in the Phase 5 cascade — matching `raw_prompt` text against domain-specific TF-IDF keywords.

## Data Flow

```
Analysis (hot path):
  prompt → heuristic_analyzer classifies domain ("saas")
  → _enrich_domain_qualifier() reads organic vocab from DomainSignalLoader cache
  → appends qualifier if 1+ keyword hit → domain_raw = "saas: growth"
  → stored on Optimization.domain_raw

Phase 5 (warm path):
  For each domain without sub-domains:
    1. Generate/refresh vocabulary via Haiku (always, no static gate)
    2. Cache in cluster_metadata["generated_qualifiers"]
    3. Push to DomainSignalLoader cache (hot path sees new vocab immediately)
    4. Three-source qualifier scan:
       Source 1: parse domain_raw qualifiers (now populated by organic vocab)
       Source 2: match intent_label against organic vocab (1+ keyword hit)
       Source 3: match raw_prompt against signal_keywords (TF-IDF, all domains)
    5. Evaluate qualifier consistency against adaptive threshold
    6. Create sub-domain if threshold crossed
```

## Invariants

- `_enrich_domain_qualifier()` never raises — returns plain domain on any error (existing contract preserved)
- `DomainSignalLoader.get_qualifiers()` never raises — returns empty dict on cache miss
- Vocabulary generation failure for one domain does not block other domains or crash Phase 5
- The three-source cascade priority order is unchanged: domain_raw > intent_label > dynamic TF-IDF
- The adaptive consistency threshold formula is unchanged: `max(0.40, 0.60 - 0.004 * members)`
- The 2-cluster breadth guard is unchanged
- The sub-domain creation flow is unchanged (state="candidate", reparenting, UMAP positioning)
- The `generated_qualifiers` metadata schema is unchanged (dict of qualifier_name → keyword_list)

## Observability

### Vocabulary Generation (Phase 5 — taxonomy event logger)

| Path | Op | Decision | Context |
|------|----|----------|---------|
| `warm` | `discover` | `vocab_generated` | `{domain, groups: int, total_keywords: int, cluster_count: int, generation_ms: float}` |
| `warm` | `discover` | `vocab_refreshed` | `{domain, reason: "cluster_count_changed"\|"first_cycle", old_groups: int, new_groups: int, generation_ms: float}` |
| `warm` | `discover` | `vocab_generation_failed` | `{domain, error: str, fallback: "cached"\|"none"}` |
| `warm` | `discover` | `vocab_cache_propagated` | `{domain, qualifier_count: int}` |
| `warm` | `discover` | `vocab_unavailable` | `{domain, reason: "generation_failed_no_cache"\|"no_clusters"}` |

Additionally, the existing `sub_domain_signal_scan` event gains a `vocab_source: "organic"` field in its context dict to make it clear the static dict is gone.

### Hot-Path Enrichment (counters on DomainSignalLoader)

No per-call event logging (hot path, runs per-optimization — too noisy). Instead, `DomainSignalLoader` tracks counters:

- `_qualifier_hits: int` — `get_qualifiers()` returned a non-empty dict
- `_qualifier_misses: int` — `get_qualifiers()` returned empty dict (domain has no vocab yet)

Exposed via `DomainSignalLoader.stats()` → wired into `GET /api/health`:

```python
{
    "qualifier_cache_hits": int,
    "qualifier_cache_misses": int,
    "domains_with_vocab": int,
    "domains_without_vocab": int,
    "last_qualifier_refresh": str | None  # ISO 8601 UTC
}
```

### TF-IDF Signal Keywords

If a domain node's `signal_keywords` is empty after extraction, log WARNING: `{domain, optimization_count, reason: "insufficient_data"|"extraction_failed"}`.

### Error Boundaries

| Component | Failure Mode | Handling |
|-----------|-------------|----------|
| Haiku vocabulary generation | LLM timeout/error | WARNING log, `vocab_generation_failed` event, use cached vocab if available, set `engine._maintenance_pending = True`, continue to next domain |
| Haiku vocabulary generation | Malformed response (not a dict, wrong schema) | WARNING log with raw response snippet (truncated to 200 chars), discard, retain cached vocab, `vocab_generation_failed` event |
| `DomainSignalLoader.get_qualifiers()` | Domain not in cache | Return empty dict, increment `_qualifier_misses` — silent, expected during cold-start |
| `DomainSignalLoader` qualifier cache refresh | DB read failure | WARNING log, retain stale cache, continue |
| `_enrich_domain_qualifier()` | Any exception | Catch-all returns plain domain unchanged, never raises (existing contract) |
| Phase 5 vocab generation loop | Single domain fails | Log, `vocab_generation_failed` event, continue to next domain (no cascade failure) |
| Phase 5 vocab generation loop | All domains fail (LLM down) | Each domain logs individually, `_maintenance_pending = True`, Phase 5 continues to qualifier evaluation with whatever cached vocab exists |

**Principle:** Vocabulary generation is best-effort. A failed generation never blocks discovery, never crashes the warm path, and never degrades the hot path. The system degrades gracefully to "no qualifier enrichment" for affected domains until the next successful generation.

## Testing Strategy

### Unit Tests

- `_enrich_domain_qualifier()` with organic vocab from `DomainSignalLoader` — verify 1-keyword-hit threshold
- `_enrich_domain_qualifier()` with empty cache (cold-start) — returns plain domain
- `DomainSignalLoader.get_qualifiers()` cache hit/miss counting
- `DomainSignalLoader.refresh_qualifiers()` updates cache and counters

### Integration Tests

- Phase 5 generates vocabulary for a domain that previously had static vocab — verify `generated_qualifiers` in metadata
- Phase 5 with Haiku failure — verify fallback to cached vocab, `_maintenance_pending` set
- Phase 5 first cycle — verify eager generation for all domains
- End-to-end: prompt classified as "saas" → enriched to "saas: growth" via organic vocab → Phase 5 counts it as domain_raw source
- `_maintenance_pending` retry after vocab generation failure — verify the next warm cycle re-attempts vocabulary generation (staleness check catches missing cache even when cluster count hasn't changed)
- Historical `domain_raw` values with old static qualifier names (e.g., "saas: growth" written by the old static vocab) — verify they are silently ignored in Source 1 when organic vocab uses different qualifier names. This is expected behavior, not an error.

### Regression Tests

- Existing sub-domain lifecycle tests still pass after `test_sub_domain_lifecycle.py` is updated to remove `_DOMAIN_QUALIFIERS` import
- Existing heuristic analyzer tests still pass after `_DOMAIN_QUALIFIERS` removal
- `_enrich_domain_qualifier()` callers unaffected (function signature unchanged)

### Data Migration Note

Existing `domain_raw` values written against the static vocabulary (e.g., "backend: auth", "database: query") may not match organic vocabulary if Haiku generates different qualifier names. This is acceptable — the static vocabulary had near-zero Source 1 hit rates for most domains (SaaS: 0/71). Future enrichments will use organic qualifiers, and Source 2 (intent_label) + Source 3 (TF-IDF) provide continuity during the transition. No data migration is needed.
